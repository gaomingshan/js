# 定时任务与自动化

> cron / systemd timer / at / anacron | 后台自动执行，运维自动化基础

---

## crontab —— 传统定时任务

### crontab 管理

```bash
crontab -e                 # 编辑当前用户的任务列表
crontab -l                 # 查看当前用户的任务
crontab -r                 # 全部删除（谨慎）
crontab -u user -e         # 编辑其他用户（root 可用）
```

### 格式 —— 分时日月周

```
* * * * *   command
─ ─ ─ ─ ─
│ │ │ │ │
│ │ │ │ └── 星期（0-7，0/7 都代表周日）
│ │ │ └──── 月份（1-12）
│ │ └────── 日期（1-31）
│ └──────── 小时（0-23）
└────────── 分钟（0-59）
```

```
特殊符号：
  *    任意           */5  每 5 单位
  ,    离散值         1,3,5
  -    范围           1-5
```

### 常用示例

```bash
# 每分钟执行一次（调试用）
* * * * * /opt/healthcheck.sh > /dev/null 2>&1

# 每天凌晨 3:00 备份
0 3 * * * /opt/backup.sh >> /var/log/backup.log 2>&1

# 每周一凌晨 2:30
30 2 * * 1 /opt/weekly-report.sh

# 每小时执行一次
0 * * * * /opt/hourly-task.sh

# 每 15 分钟
*/15 * * * * /opt/check.sh

# 工作日（周一到周五）早 9 点
0 9 * * 1-5 /opt/start-work.sh

# 每月 1 号凌晨 4 点
0 4 1 * * /opt/monthly-clean.sh

# 重启后执行一次
@reboot /opt/startup.sh
```

### 环境变量陷阱

cron 执行的环境和登录 shell 不同，**PATH 极短**：

```bash
crontab -l
# 默认 PATH=/usr/bin:/bin
# 脚本中不要依赖 ~/.bashrc 的配置

# 解决办法 1：脚本内写全路径
0 3 * * * /usr/local/bin/python3 /opt/script.py

# 解决办法 2：脚本开头 export PATH
# 在脚本内写 export PATH="/usr/local/bin:$PATH"

# 解决办法 3：crontab 内设 PATH
PATH=/usr/local/bin:/usr/bin:/bin
0 3 * * * python3 /opt/script.py
```

### 日志

```bash
# crond 自己输出到系统日志
journalctl -u crond -f

# 脚本输出必须手动重定向
# 2>&1 表示 stderr 合并到 stdout
0 3 * * * /opt/backup.sh >> /var/log/backup.log 2>&1

# 不想收邮件（默认会发 MAILTO）
MAILTO=""
```

### 安全控制

```bash
/etc/cron.allow      # 白名单——只有这里的用户可以建任务
/etc/cron.deny       # 黑名单——禁止谁建任务
# 如果 allow 存在，只检查 allow；都不存在则 root 可用
```

## systemd timer —— cron 的现代化替代

cron 的缺点：不感知 systemd、不记录日志、环境变量简略、缺触发事件机制。systemd timer 解决这些问题。

### 基本结构：一对文件

```ini
# /etc/systemd/system/backup.service
[Service]
ExecStart=/opt/backup.sh
Type=oneshot

# /etc/systemd/system/backup.timer
[Unit]
Description=每日备份
[Timer]
OnCalendar=daily
Persistent=true          # 错过执行时间后立即补跑（机器关机期间的任务）
RandomizedDelaySec=300   # 延迟 0-5 分钟，防雪崩
[Install]
WantedBy=timers.target
```

### 操作

```bash
systemctl enable --now backup.timer   # 启用并启动
systemctl start backup.timer
systemctl list-timers                  # 查看所有定时器
systemctl status backup.timer
systemctl list-timers --all            # 包含已触发的
journalctl -u backup.service           # 看执行日志
```

### OnCalendar 格式

```
Mon/Wed/Fri          星期名
daily                每天凌晨
hourly               每小时
*-*-* 00:00:00       YYYY-MM-DD HH:MM:SS
Mon..Fri 09:30:00    周一到周五 9:30
*-*-1/2 00:00:00     隔月 1 号
```

### cron vs systemd timer

| 对比 | cron | systemd timer |
|------|------|---------------|
| 精度 | 分钟级 | 秒级 |
| 日志 | 需手动重定向 | journalctl 统一 |
| 环境变量 | PATH 极短 | 继承 service 环境 |
| Persistent | ❌ 关机错过就丢了 | ✅ Persistent=true 补跑 |
| 随机延迟 | ❌ 手动 sleep | ✅ RandomizedDelaySec |
| 依赖关系 | ❌ | ✅ After/Requires |
| 触发器 | 仅时间 | 时间+事件（如启动后/设备插入后） |

## at —— 一次性定时任务

```bash
echo "/opt/upgrade.sh" | at now + 30 minutes
at now + 2 hours
at> systemctl restart nginx
at> 按 Ctrl+D 结束

# 指定时间
at 03:00
at 14:30 tomorrow
at 10:00 next week
at 23:00 2026-12-31

atq                     # 查看等待中的任务列表
atrm 5                  # 删除编号为 5 的任务
at -c 5                 # 查看编号 5 的具体内容
```

```bash
# 限制 at 使用
/etc/at.allow
/etc/at.deny
```

## anacron —— 补偿掉电漏跑

crond 假设机器 7×24 运行。笔记本/桌面机关机后，cron 任务直接错过。anacron 用来**补偿**——记录上次执行时间，开机后检查是否该跑。

```bash
# /etc/anacrontab
# 天数  延迟(分)  job-id         命令
1      10        daily.backup   /opt/backup.sh
7      30        weekly.clean   /opt/cleanup.sh

# 含义：每天（1天）执行一次，如果上次执行超过 1 天，等 10 分钟后跑
# job-id 是唯一标识，保存在 /var/spool/anacron/ 下
```

| 对比 | cron | anacron |
|------|------|---------|
| 假设 | 一直开机 | 可能关机 |
| 精度 | 分钟级 | 天级 |
| 补偿 | ❌ 错过就错过 | ✅ 开机后补跑 |
| 场景 | 服务器 | 桌面/笔记本 |

## 实践场景

### 场景 1：每日数据库备份（全量+保留策略）

```bash
#!/bin/bash
# /opt/backup-mysql.sh
DB_NAME=myapp
BACKUP_DIR=/data/backup/mysql
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=7

mkdir -p "$BACKUP_DIR"
mysqldump --single-transaction "$DB_NAME" | gzip > "$BACKUP_DIR/${DB_NAME}_${DATE}.sql.gz"

# 删 7 天前的旧备份
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete

echo "$(date) backup done, size: $(du -sh $BACKUP_DIR/${DB_NAME}_${DATE}.sql.gz | cut -f1)" >> /var/log/backup-mysql.log
```

```bash
# crontab：每天凌晨 2 点执行
0 2 * * * /opt/backup-mysql.sh >> /var/log/backup-mysql.log 2>&1
```

### 场景 2：定时健康检查+告警

```bash
#!/bin/bash
# /opt/healthcheck.sh
# 检查项：磁盘 > 90%、内存 < 200M、关键进程存活

ADMIN=admin@example.com

# 磁盘
DISK_USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')
if [ "$DISK_USAGE" -gt 90 ]; then
  echo "磁盘 / 使用率 ${DISK_USAGE}%" | mail -s "Disk Alert" $ADMIN
fi

# 内存
FREE_MEM=$(free -m | awk 'NR==2 {print $7}')
if [ "$FREE_MEM" -lt 200 ]; then
  echo "可用内存仅 ${FREE_MEM}M" | mail -s "Memory Alert" $ADMIN
fi

# 进程
for proc in nginx mysql redis-server; do
  if ! pgrep -x "$proc" > /dev/null; then
    echo "$proc 挂了" | mail -s "Process Alert: $proc" $ADMIN
  fi
done
```

### 场景 3：用 systemd timer 替代 cron

当需要以下特性时，优先用 systemd timer 而非 cron：

- 任务依赖某个服务就绪（如 After=network.target）
- 机器可能关机，需要补跑（Persistent=true）
- 多个凌晨任务需要错峰（RandomizedDelaySec）
- 需要对任务做 cgroup 资源限制（CPU/MemoryLimit=）
- 希望日志统一走 journalctl

```bash
# 模板
cat > /etc/systemd/system/cleanup.service <<'SVC'
[Service]
Type=oneshot
ExecStart=/opt/cleanup.sh
SVC

cat > /etc/systemd/system/cleanup.timer <<'TIMER'
[Unit]
Description=每日清理
[Timer]
OnCalendar=03:00
Persistent=true
RandomizedDelaySec=600
[Install]
WantedBy=timers.target
TIMER

systemctl daemon-reload && systemctl enable --now cleanup.timer
```
