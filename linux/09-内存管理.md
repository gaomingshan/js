# 内存管理

> 查看/缓存/Swap/OOM Killer | 泄漏排查/性能调优 | 物理内存 + 虚拟内存

---

## 内存查看

### free —— 最常用

```bash
free -h               # 总/用/空/缓存（人类可读）
free -h -w            # 分 active / inactive
free -s 2             # 每 2 秒刷新
```

输出解读：

```
              total        used        free      shared  buff/cache   available
Mem:           31G         12G         1G         200M         18G         18G
Swap:           2G          0G         2G
```

| 列 | 含义 |
|----|------|
| `total` | 物理内存总量 |
| `used` | 被占用的（= total - free - buff/cache） |
| `free` | 完全空闲 |
| `buff/cache` | 内核缓存（可回收） |
| `available` | 可用估算（含可回收缓存） |

### vmstat —— 内存 + 交换

```bash
vmstat 2              # 每 2 秒输出
```

关键列：

| 列 | 含义 | 警戒 |
|----|------|------|
| `free` | 空闲内存（KB） | 持续偏低 |
| `si` | 从磁盘换入（KB/s） | > 0 说明内存不足 |
| `so` | 换出到磁盘（KB/s） | > 0 说明内存不足 |
| `cache` | Page Cache | — |
| `buffer` | Buffer Cache | — |

### top / htop —— 按内存排序

```bash
top -o %MEM           # 按内存占用排序
htop                  # 交互式，F6 按 MEM%
```

`top` 内存列说明：

| 列 | 含义 |
|----|------|
| `VIRT` | 虚拟内存（含共享库 + 映射） |
| `RES` | 常驻物理内存（RSS） |
| `SHR` | 共享内存 |
| `MEM%` | RES / 物理总量 |

### /proc/meminfo —— 详细内存信息

```bash
cat /proc/meminfo
cat /proc/meminfo | grep -E '^(Cached|Dirty|Writeback|PageTables|AnonPages|Slab|SReclaimable)'
```

关键字段：

| 字段 | 含义 |
|------|------|
| `MemTotal` | 物理内存总量 |
| `MemFree` | 完全空闲 |
| `MemAvailable` | 可用（含可回收缓存） |
| `Cached` | Page Cache + tmpfs |
| `Dirty` | 脏页（待写盘） |
| `Writeback` | 正在写盘 |
| `AnonPages` | 匿名页（堆/栈） |
| `PageTables` | 页表所占内存 |
| `Slab` | 内核 slab |
| `SReclaimable` | 可回收 slab |

## 缓存机制

### Page Cache

文件内容缓存，读文件时内核自动缓存，写文件时延迟写盘。

```bash
# 缓存占用查看
cat /proc/meminfo | grep -E '^(Cached|Dirty|Writeback)'

# 清缓存（仅测试环境！）
echo 1 > /proc/sys/vm/drop_caches    # 清 Page Cache
echo 2 > /proc/sys/vm/drop_caches    # 清 slab（dentry + inode）
echo 3 > /proc/sys/vm/drop_caches    # 清全部
```

**生产环境不要清缓存。** 缓存的目的是加速，清掉只会降低性能。

### dentry_cache / inode_cache

文件系统元数据缓存，属于 Slab。

```bash
# 查看 slab 分布
slabtop -o            # 按对象大小排序
cat /proc/slabinfo    # 原始数据

# dentry 缓存太大时可调
sysctl -w vm.vfs_cache_pressure=200   # 默认 100，越大回收越积极
```

### 内存分配路径

```
应用 malloc() → glibc → brk()/mmap()
  → 内核缺页中断 → 分配物理页 → Page Cache / AnonPages
  → 内存不足 → 内核回收（kswapd / direct reclaim）→ 可能触发 OOM
```

## Swap

### 查看 Swap

```bash
swapon --show         # 查看 swap 设备和用量
free -h               # 看 Swap 行
cat /proc/swaps       # 同上，更详细
```

### 创建 Swap

```bash
# 分区方式
mkswap /dev/sdb3 && swapon /dev/sdb3

# 文件方式（不需要分区）
dd if=/dev/zero of=/swapfile bs=1M count=4096
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile

# fstab 持久化
# /swapfile  none  swap  sw  0  0
```

### Swap 调优

```bash
# swappiness：值越小越倾向物理内存，越大越倾向 swap
sysctl vm.swappiness=10           # 服务器推荐 10，默认 60
# 永久
echo "vm.swappiness=10" >> /etc/sysctl.conf

# Swap 关闭（建议先确保物理内存充裕）
swapoff -a                        # 关所有 swap
swapoff /dev/sdb3                 # 关指定
```

### Swap 飙高排查

```bash
# 确认是物理内存不足还是高 swappiness
free -h
vmstat 1                    # si/so 非零说明在 swap

# 谁在大量占用内存
top -o %MEM
ps aux --sort=-%mem | head -10
```

## OOM Killer

### 查看 OOM 记录

```bash
dmesg | grep -i "killed process"
dmesg | grep -i "oom"
journalctl -k | grep -i oom        # systemd 日志
```

输出示例：
```
[12345.678] oom-killer: gfp_mask=0x...
[12345.679] Out of memory: Killed process 1234 (java)
[12345.679] total-vm:..., anon-rss:..., file-rss:...
```

### 查看/调整进程 OOM 评分

```bash
cat /proc/1234/oom_score            # 当前评分（0-1000，越高越容易被杀）
cat /proc/1234/oom_score_adj        # 用户可调偏移（-1000 ~ 1000）

# 保护关键进程不被 OOM 杀掉
echo -1000 > /proc/1234/oom_score_adj

# 让某进程更可能被优先杀掉
echo 500 > /proc/1234/oom_score_adj
```

评分规则：

| 因素 | 影响 |
|------|------|
| RSS 越大 | 评分越高 |
| oom_score_adj | 调整值 |
| 子进程数 | 累加 |
| root 权限进程 | 略有降低 |

## 排障

### 内存泄漏

```bash
# 1. 确认泄漏
top -o %MEM                  # RES 持续增长

# 2. 看进程内存映射
pmap -x PID                  # 看各段（heap、anon 段大小变化）
cat /proc/PID/smaps          # 逐段详细

# 3. 堆内泄漏 → jmap / gcore
jmap -heap PID               # Java
jmap -histo:live PID         # 存活对象分布
gcore PID                    # 全 dump，下来分析

# 4. 堆外泄漏（JNI / DirectBuffer）
pmap -x PID | grep anon      # 匿名映射段
cat /proc/PID/status | grep VmRSS    # RSS
```

### 内存不足的典型现象

| 现象 | 可能原因 |
|------|----------|
| `free -h` 显示可用少 | 正常（缓存多） |
| `vmstat` si/so > 0 | 物理内存不足 |
| 进程被 kill | OOM Killer |
| 应用响应变慢 | 大量 swap + 磁盘 IO |
| `dmesg` 有 "page allocation failure" | 内存碎片 |

### 内存碎片

```bash
# 查看可用连续大页
cat /proc/buddyinfo
# 看 DMA / Normal 区的 order-0~9 可用页数

# 大页优化（HugePages）
echo 1024 > /proc/sys/vm/nr_hugepages
cat /proc/meminfo | grep Huge
```

## 实战场景

### 场景 1：定位 Java 堆外内存泄漏

```bash
# 1. 现象：RES 不断增长，堆内 dump 正常
top -o %MEM
jstat -gc PID                # 堆内占比不大

# 2. 排查匿名映射
pmap -x PID | sort -k3 -rn | head -20
# 看 "anon" 段是否异常增大

# 3. 检查 DirectBuffer
jcmd PID VM.native_memory summary scale=MB
# NMT 开启需要 -XX:NativeMemoryTracking=detail

# 4. 常见元凶：DirectByteBuffer、JNI、netty、gRPC
```

### 场景 2：保护关键进程不被 OOM 杀掉

```bash
# 1. 查看哪些进程容易被杀
for pid in $(pgrep -f java); do
  echo "PID $pid score=$(cat /proc/$pid/oom_score)"
done

# 2. 保护数据库 / 核心服务
echo -1000 > /proc/$(pgrep -f mysqld)/oom_score_adj

# 3. 永久（systemd service）
# [Service]
# OOMScoreAdjust=-1000

# 4. 限制可杀进程的内存上限
systemctl set-property user.slice MemoryMax=10G
```

### 场景 3：关闭不必要的 swap

```bash
# 确认物理内存足够
free -h

# 临时关闭
swapoff -a

# 永久关闭（注释 fstab）
sed -i '/swap/d' /etc/fstab

# 或直接删 swap 文件
rm -f /swapfile

# 确认
swapon --show    # 应无输出
```

### 场景 4：定位谁把内存吃光了

```bash
# 1. 总览
free -h
cat /proc/meminfo | head -20

# 2. 进程排行
ps aux --sort=-%rss | head -10

# 3. slab 缓存是否异常
slabtop -o | head -10
# 如果 dentry 或 inode_cache 过大
sysctl -w vm.vfs_cache_pressure=500   # 加速回收

# 4. Page Cache 是否过大
echo 3 > /proc/sys/vm/drop_caches     # 仅测试环境
```

### 场景 5：HugePages 优化数据库

```bash
# 1. 计算需要的大页数（MySQL 为例）
# 假设 innodb_buffer_pool_size=8G，每页 2M
# 8 * 1024 / 2 = 4096 页

# 2. 配置
echo 4096 > /proc/sys/vm/nr_hugepages
# 或写入 /etc/sysctl.conf
echo "vm.nr_hugepages=4096" >> /etc/sysctl.conf

# 3. 确认
cat /proc/meminfo | grep -i huge
# HugePages_Total: 4096
# HugePages_Free:  0   ← 全部被使用
```
