# Parcel 零配置理念

## 概述

Parcel 的核心主张可以概括为：**Zero Config（零配置）**。

它试图把构建工具从“需要你搭一套系统”变成“你只要把项目写好，剩下交给工具”。

这背后有两个关键设计点：

- **自动依赖分析**：从入口文件出发，识别 JS/CSS/HTML/资源的依赖关系
- **约定优于配置**：内置大量默认规则与插件管线，减少显式配置

---

## 一、Parcel 解决了什么问题

### 1.1 快速启动项目

对于新手或小团队来说，最大痛点往往不是“构建性能极限”，而是：

- 不知道该配哪些 loader/plugin
- 配完后不知道哪里错了

Parcel 的目标是让你：

- 有一个入口（HTML/JS）
- 直接跑 dev server
- 自动支持常见资源类型与转换

### 1.2 统一处理多种资源类型

Parcel 倾向于把“资源类型识别 + 转换”作为内置能力：

- JS/TS/JSX
- CSS/PostCSS
- 图片/字体/SVG
- HTML 作为真实入口

> **关键点**
>
> Parcel 的哲学是“你写代码 → 我推断你要什么”，而不是“你告诉我每一步怎么做”。

---

## 二、Parcel 的工作方式（直觉模型）

### 2.1 从入口构建依赖图

当你以 HTML 作为入口时：

```html
<link rel="stylesheet" href="./styles.css" />
<script type="module" src="./main.ts"></script>
```

Parcel 会把这些资源都纳入构建图，并自动决定对应的处理管线。

### 2.2 内置管线与缓存

现代构建工具几乎都依赖缓存与增量构建。

Parcel 的零配置体验成立的前提之一是：

- 默认的处理管线足够覆盖主流需求
- 缓存/增量策略默认是“合理的”

---

## 三、Parcel 的优势

### 3.1 上手成本低

- 适合新手理解“构建工具能做什么”
- 适合快速原型与中小项目

### 3.2 默认值覆盖面广

- 你不需要先学一堆 loader/plugin 名字
- 常见场景能直接工作

---

## 四、Parcel 的局限（不擅长什么）

### 4.1 深度定制的成本更高

当项目进入“需要严格控制产物策略”的阶段，你可能会遇到：

- 很多默认行为是自动推断，改起来不如显式配置直观
- 团队要形成统一规范时，需要额外约束与工具化

### 4.2 生态与企业级沉淀相对弱一些

- 与 webpack 生态相比，很多企业内部/社区插件沉淀更少
- 复杂历史场景（老包袱）可能需要额外方案

> **建议**
>
> Parcel 很适合作为“快速启动 + 学习构建工具”的路径；当你需要更强产物控制或复用大量生态时，再评估 Vite/webpack/Rsbuild。
