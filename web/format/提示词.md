你是一名资深前端架构师与技术教育者，在当前目录下，请生成面向「后端开发者」的前端「代码规范与工程化」系统化学习内容及面试题，要求如下：

【学习对象】
- 以 Java / Go / Node.js 等后端背景为主的开发者
- 已能阅读和编写基础前端代码，但希望进一步：
  - 从“工程系统”的角度理解前端代码规范
  - 建立类似后端架构规范、编码约束、工程治理的认知体系
  - 理解前端代码在“多人协作 + 长期演进”中的失控风险与治理手段

【学习目标】
- 建立前端代码规范与工程化的整体技术地图，而非零散规则
- 系统理解前端工程中：
  - 代码规范存在的根本原因
  - 规范如何影响可维护性、可扩展性与团队效率
  - 工具链如何落地和“强制执行”规范
- 能在真实项目与团队中：
  - 制定合理的前端代码规范
  - 判断哪些规范是“必要约束”，哪些是“过度设计”

【生成顺序与文件结构】
1. 生成大纲文件 content-outline.md，每个小节带有 Markdown 跳转链接到对应 content 文件
2. 生成教学内容，存放于 content/ 目录
3. 生成面试题汇总，存放于 quiz/quiz.md

【文件命名规则】
- 大纲文件：content-outline.md
- 教学内容：content/content-1.md、content/content-2.md … 按大纲顺序递增
- 面试题汇总：quiz/quiz.md
- 链接格式：[小节标题](./content/content-x.md)，x 按大纲顺序递增
- 统一使用 Markdown，中文，技术严谨，面向工程与团队实践

【教学内容要求】
- 内容必须以“前端工程化与代码治理”为第一目标，而不是工具速查
- 核心内容必须围绕：
  - 前端代码规范的设计目标与分类体系
  - JavaScript / TypeScript / HTML / CSS 的职责边界与协作规则
  - 代码风格、结构规范与可维护性的关系
  - 工程化工具在规范执行中的角色（而非工具用法本身）
  - 规范在 Monorepo、多团队协作、长期项目中的演进问题
- 需要显式对标后端工程经验：
  - 前端代码规范 ≈ 后端编码规范 / 架构约束
  - Lint / Format ≈ 编译期 / 静态分析
  - 工程约束 ≈ 服务治理与技术规范

【每小节必须包含】
- 核心概念说明（是什么）
- 设计动机 / 工程原理（为什么需要 & 如何约束）
- 常见误区或边界问题（后端转前端高频踩坑）
- 工程或团队实践示例（真实项目 / 多人协作场景）

【难度分级】
- 基础：
  - 前端代码规范的必要性、问题来源、基本分类
- 进阶：
  - 不同语言与层级的规范边界（JS / TS / HTML / CSS）
  - 规范与工具链的结合方式
- 高级：
  - 大型项目、Monorepo、跨团队下的规范治理
  - 规范演进、技术债控制与架构约束

【面试题要求】
- 单选 / 多选 / 代码分析 / 场景决策题结合，约 100 题
- 难度递进：基础认知 → 原理理解 → 架构与治理决策
- 覆盖范围必须与教学大纲完全一致
- 题目必须强调：
  - 代码规范为什么要这样设计
  - 不同规范策略的取舍逻辑
  - 工具自动化与人工约束的边界
- 每题提供答案与解析
- 解析格式要求：
  - 结论 → 原理 → 易错点 / 反例
  - 超过 15 行 Markdown 必须拆分
- 面试题仅生成一个汇总文件 quiz/quiz.md

【质量红线】
- 禁止只讲 ESLint / Prettier / 工具配置，不讲工程动机
- 禁止罗列规则而不解释其治理目标
- 禁止脱离团队协作与长期维护谈代码规范
- 禁止把代码规范等同于“个人风格偏好”
- 概念讲不清时宁可删减，不可堆字

【额外说明】
- 明确对比前端与后端工程规范：
  - 前端代码规范 vs 后端编码规范
  - 前端工程化 vs 构建 / CI / 质量门禁
- 教学内容必须覆盖真实工程问题：
  - 代码风格混乱导致的维护成本
  - 规范失效或过度严格带来的反效果
  - 多项目 / 多团队下规范难以统一的问题
- 大纲文件必须生成 Markdown 链接，格式为 [小节标题](./content/content-x.md)，确保点击可跳转
