> 不讲不讲百科式知识点，只讲"这个东西对你写代码有什么用"

## 第一部分：数据库架构全貌 — 先在心里画一张地图
**对日常有什么用**：以后看任何数据库问题，脑子都有完整的"请求路径"

- 一条SQL的一生：连接器 → 查询解析 → 语法/语义分析 → 优化器 → 执行器 → 存储引擎
- InnoDB架构速写：Buffer Pool、Redo Log Buffer、Change Buffer、Adaptive Hash Index
- 三大数据库思维模型（MySQL/PostgreSQL/Oracle 的定位差异）
- WAL机制 + Checkpoint + 崩溃恢复流程

## 第二部分：SQL执行模型 — 你写的不只是"代码"
**对日常有什么用**：理解"数据库怎么跑这段SQL"，才能写出高效的SQL

- SELECT 执行顺序（FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT）
- 理解了执行顺序，你就知道为什么别名在WHERE不能用、为什么WHERE里不能有聚合函数
- DML的底层的读-改-写流程
- DDL事务差异（MySQL隐式提交 vs PostgreSQL事务型DDL）
- JOIN本质与算法选择（NLJ → Hash Join → Merge Join）
- 子查询 vs JOIN vs CTE 的选择逻辑

## 第三部分：字段类型 — 每个选择都有代价
**对日常有什么用**：表设计时能做有意识的选择

- 数值陷阱：INT(11)的11没意义、DECIMAL精度、自增用完了怎么办
- 字符代价：VARCHAR(255)为什么常见、TEXT的隐藏性能代价
- 时间选择：DATETIME vs TIMESTAMP 核心区别
- JSON类型：什么时候用、什么时候别用
- **核心思维**：字段选择 = 存储空间 × 查询性能 × 使用便利性的权衡

## 第四部分：数据存储 — 一行数据到底存在哪
**对日常有什么用**：理解"数据怎么放"，才能理解"索引为什么快"

- InnoDB行格式（变长字段、NULL列表、行溢出）
- 页结构：16KB页、页目录、B+Tree叶子/非叶子节点
- 一页能存多少行 → 树高 → IO次数（核心公式）
- 聚簇索引 = InnoDB表本身：主键选错=写入性能崩了
- 这条WHERE id = N，B+Tree从根到叶经历了什么

## 第五部分：索引（全文核心 — 分六节）
**对日常有什么用**：90%的性能问题出在这里

### 5.1 索引的本质与类型
- B+Tree为什么适合数据库（IO优化 + 范围查询 + 有序性）
- Hash索引：为什么InnoDB不支持显式创建
- 全文索引、空间索引：知道存在就够了

### 5.2 聚簇索引 vs 辅助索引
- 回表 = 随机IO（索引优化的核心矛盾）
- 覆盖索引（Using Index / Using Index Condition 的区别）
- 索引下推 ICP

### 5.3 联合索引与最左前缀
- 不是"规则"，是B+Tree的有序性决定的
- (a, b, c) 能命中哪些查询？一张图讲清楚
- 误区：三个单列索引 ≠ 一个联合索引

### 5.4 索引创建与代价
- 空间代价：每个索引是一棵B+Tree
- 写入代价：INSERT/UPDATE要维护所有索引
- 索引不是越多越好，是越精准越好

### 5.5 索引失效
- 类型转换、函数操作、LIKE前缀、OR、NOT...
- 这些失效的本质都是"破坏了索引的有序性"

### 5.6 索引优化策略
- Cardinality + 索引选择性 = 该不该建索引
- Index Merge、MRR（从随机IO变顺序IO）
- 碎片产生与重建

## 第六部分：查询优化器 — 你的SQL是怎么被"分析"的
**对日常有什么用**：能读懂执行计划，能判断"优化器选对了没"

- CBO核心：基于成本（IO+CPU）选计划，不是"看着对不对"
- 统计信息是优化器的眼睛——过期了SQL就会突然变慢
- 执行计划关键指标：type（const → ref → range → index → ALL）、Extra
- 优化器"选错"的常见原因
- Hint：1%的场景才需要用

---

## 第七部分：事务与并发 — 你的SQL为什么卡住了
**对日常有什么用**：遇到锁等待、死锁、数据不一致时能定位

- ACID实现：Undo Log（原子性）+ Redo Log（持久性）+ MVCC（隔离性）
- MVCC：Read View + 版本链，快照读不加锁
- 隔离级别：RR vs RC，InnoDB用间隙锁解决幻读
- 死锁本质 + 读懂死锁日志
- 只讲InnoDB实现，Oracle/PG补充差异
