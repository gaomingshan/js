# 分布式架构模型

## 概述

Elasticsearch 从设计之初就是为分布式场景打造的，通过集群（Cluster）、节点（Node）、分片（Shard）的协作，实现数据的水平扩展、高可用和负载均衡。理解分布式架构是掌握 ES 的核心。

## 核心组件关系

```
集群（Cluster）
  ├── 节点1（Node）
  │     ├── 索引A的主分片0
  │     ├── 索引A的副本分片1
  │     └── 索引B的主分片2
  ├── 节点2（Node）
  │     ├── 索引A的主分片1
  │     ├── 索引A的副本分片0
  │     └── 索引B的副本分片2
  └── 节点3（Node）
        ├── 索引B的主分片0
        ├── 索引B的主分片1
        └── 索引A的主分片2
```

## 集群（Cluster）

### 定义

**集群是一个或多个节点的集合**，共同持有全部数据，提供索引和搜索功能。

### 集群标识

每个集群有唯一的名称（默认 `elasticsearch`）：

```yaml
# elasticsearch.yml
cluster.name: my-application
```

**重要性**：
- 节点通过集群名加入集群
- 不同集群名的节点不会互相通信
- 生产环境必须修改默认集群名

### 集群状态

通过 `_cluster/health` API 查看集群健康状态：

```json
GET /_cluster/health

{
  "cluster_name": "my-application",
  "status": "green",              // 健康状态
  "timed_out": false,
  "number_of_nodes": 3,            // 节点数
  "number_of_data_nodes": 3,       // 数据节点数
  "active_primary_shards": 10,     // 活跃主分片数
  "active_shards": 20,             // 活跃总分片数
  "relocating_shards": 0,          // 正在迁移的分片
  "initializing_shards": 0,        // 正在初始化的分片
  "unassigned_shards": 0           // 未分配的分片
}
```

### 集群健康状态

| 状态 | 含义 | 说明 |
|------|------|------|
| **Green** | 健康 | 所有主分片和副本分片都已分配 |
| **Yellow** | 警告 | 所有主分片已分配，但部分副本分片未分配 |
| **Red** | 异常 | 部分主分片未分配，数据不完整 |

**状态场景**：

```
Green：生产环境的目标状态
  - 所有功能正常
  - 数据完整且有冗余

Yellow：单节点集群的常见状态
  - 可以正常读写
  - 无数据冗余，存在风险
  - 示例：单节点集群无法分配副本分片

Red：严重问题，需立即处理
  - 部分数据不可访问
  - 查询可能返回不完整结果
  - 示例：节点宕机且未恢复主分片
```

## 节点（Node）

### 定义

**节点是集群中的单个服务器实例**，存储数据并参与集群的索引和搜索。

### 节点标识

每个节点有唯一的名称和 ID：

```yaml
# elasticsearch.yml
node.name: node-1

# 启动后自动生成 UUID
node.id: "FZT9Q3vXQJ2K8..."
```

### 节点角色

Elasticsearch 7.x+ 引入了细粒度的节点角色：

```yaml
# elasticsearch.yml
node.roles: [ master, data, ingest, ml ]
```

#### 1. Master 节点（主节点）

**职责**：管理集群级别的操作

```yaml
node.roles: [ master ]
```

**核心功能**：
- 创建/删除索引
- 管理集群拓扑（节点加入/离开）
- 分片分配决策
- 维护集群状态（Cluster State）

**选主机制**：
- 集群中有多个 Master-eligible 节点
- 通过选举产生一个活跃的 Master 节点
- Master 节点故障时，重新选举

**最佳实践**：
- 生产环境配置 3 个专用 Master 节点（奇数个避免脑裂）
- Master 节点不存储数据，专注于集群管理

#### 2. Data 节点（数据节点）

**职责**：存储数据并执行 CRUD、搜索、聚合操作

```yaml
node.roles: [ data ]
```

**核心功能**：
- 存储分片数据
- 执行查询、索引、聚合
- 消耗大量 CPU、内存、I/O

**最佳实践**：
- 根据数据量和查询负载配置数据节点数量
- 数据节点需要高性能硬件（SSD、大内存）

#### 3. Coordinating 节点（协调节点）

**职责**：接收客户端请求，分发到数据节点，聚合结果

```yaml
# 不配置任何角色即为纯协调节点
node.roles: [ ]
```

**核心功能**：
- 路由请求到正确的节点
- 聚合各节点返回的结果
- 不存储数据，不参与选主

**使用场景**：
- 大规模集群的负载均衡层
- 聚合查询较多的场景

**注意**：所有节点都具备协调能力，专用协调节点是可选的优化。

#### 4. Ingest 节点（预处理节点）

**职责**：在索引文档前进行数据预处理

```yaml
node.roles: [ ingest ]
```

**核心功能**：
- 执行 Ingest Pipeline
- 数据转换、富化、解析

**使用场景**：
- 日志解析（Grok）
- 数据格式转换
- IP 地理位置解析

#### 5. Machine Learning 节点

**职责**：执行机器学习任务

```yaml
node.roles: [ ml ]
```

**核心功能**：
- 异常检测
- 数据分析
- 需要大量 CPU 和内存

### 节点角色组合

**常见配置**：

```yaml
# 小型集群（< 10 节点）：混合节点
node.roles: [ master, data, ingest ]

# 大型集群：专用角色
# Master 节点
node.roles: [ master ]

# Data 节点
node.roles: [ data, ingest ]

# 协调节点
node.roles: [ ]
```

## 分片（Shard）

### 分片的本质

**分片是索引的物理分割**，每个分片本质上是一个独立的 Lucene 索引。

```
索引：products（1000万文档，50GB）
  ↓ 分为 5 个主分片
  
主分片0：200万文档，10GB
主分片1：200万文档，10GB
主分片2：200万文档，10GB
主分片3：200万文档，10GB
主分片4：200万文档，10GB
```

### 分片路由算法

**文档如何确定存储在哪个分片？**

```java
shard_num = hash(_routing) % num_primary_shards
```

**默认路由**：使用文档的 `_id` 作为路由值

```json
PUT /products/_doc/12345
{
  "name": "iPhone"
}

// 计算分片：hash("12345") % 5 = 2
// 存储到主分片 2
```

**自定义路由**：指定路由字段

```json
PUT /products/_doc/12345?routing=user123
{
  "name": "iPhone",
  "user_id": "user123"
}

// 计算分片：hash("user123") % 5 = 1
// 同一用户的文档存储在同一分片，优化查询
```

**重要性**：
- 主分片数量确定后不可修改（路由算法依赖主分片数）
- 修改主分片数会导致路由失效

### 主分片与副本分片

#### 主分片（Primary Shard）

**定义**：原始数据的唯一持有者，负责写入操作。

```
写入流程：
  客户端 → 协调节点 → 主分片 → 副本分片
```

**特性**：
- 数量在索引创建时确定，后续不可修改
- 每个文档只属于一个主分片

#### 副本分片（Replica Shard）

**定义**：主分片的完整副本，提供数据冗余和读性能扩展。

```
副本数量：可动态调整

PUT /products/_settings
{
  "number_of_replicas": 2
}
```

**特性**：
- 副本分片不会与对应的主分片分配在同一节点
- 主分片故障时，副本可提升为主分片
- 查询请求可分发到主分片或副本分片

### 分片分配示例

**配置**：
- 3 个节点
- 索引：3 个主分片，2 个副本
- 总分片数：3 (primary) + 6 (replica) = 9

**分配结果**：

```
节点1：
  - P0（主分片0）
  - R1（副本分片1）
  - R2（副本分片2）

节点2：
  - P1（主分片1）
  - R0（副本分片0）
  - R2（副本分片2）

节点3：
  - P2（主分片2）
  - R0（副本分片0）
  - R1（副本分片1）
```

**原则**：
- 副本分片与主分片不在同一节点
- 分片均匀分布在各节点

## 水平扩展与垂直扩展

### 水平扩展（Scale Out）

**增加节点数量**，分散数据和负载。

```
3节点 → 6节点

分片重新分配：
  节点1：P0, R1
  节点2：P1, R2
  节点3：P2, R0
  节点4：R1, R2
  节点5：R0, P0
  节点6：P1, P2
```

**优势**：
- 增加存储容量
- 提升查询并发能力
- 提高可用性

**限制**：
- 主分片数量限制扩展能力
- 示例：3 个主分片，最多 3 个节点能存储主分片

### 垂直扩展（Scale Up）

**升级节点硬件**：更多内存、更快的 CPU、SSD。

**适用场景**：
- 单个分片过大
- 查询性能瓶颈

**限制**：
- 单机硬件有上限
- 成本高

### 最佳实践

**推荐**：水平扩展为主，垂直扩展为辅。

```
规划分片数量 = 预期数据量 / 单分片目标大小（20-40GB）

示例：
  预期数据：500GB
  单分片：30GB
  主分片数 = 500 / 30 ≈ 17
```

## 集群容错与故障转移

### 节点故障场景

#### 场景1：数据节点故障

```
初始状态：
  节点1: P0, R1
  节点2: P1, R0
  节点3: P2, R2

节点1故障：
  丢失 P0 和 R1

集群反应：
  1. 检测到节点1离线
  2. 提升 R0（节点2）为新的 P0
  3. 在节点3创建新的 R0
  
最终状态：
  节点2: P1, P0（提升）
  节点3: P2, R2, R0（新建）
  
集群状态：Yellow（缺少一个副本）
```

#### 场景2：Master 节点故障

```
初始状态：
  节点1: Master
  节点2: Master-eligible
  节点3: Master-eligible

节点1故障：
  集群无主节点

集群反应：
  1. 节点2、节点3发起选举
  2. 选举出新的 Master（如节点2）
  3. 新 Master 管理集群
  
选举时间：通常 < 1 秒
```

### 脑裂问题（Split-Brain）

**定义**：网络分区导致集群分裂成多个子集群，每个子集群都有自己的 Master。

```
原始集群：
  节点1（Master）、节点2、节点3

网络分区：
  子集群1：节点1（Master）
  子集群2：节点2（选举新Master）、节点3

问题：
  - 两个 Master 同时存在
  - 数据不一致
  - 网络恢复后数据冲突
```

**解决方案**：配置最小主节点数（Quorum）

```yaml
# elasticsearch.yml (7.0 之前)
discovery.zen.minimum_master_nodes: 2

# 公式：(master_eligible_nodes / 2) + 1
# 3 个节点：(3 / 2) + 1 = 2
```

**7.0+ 自动处理**：ES 7.0+ 自动计算 quorum，无需手动配置。

### 高可用架构建议

```
最小高可用配置：
  - 3 个 Master-eligible 节点（避免脑裂）
  - 至少 2 个副本
  - 跨机架/可用区部署

生产级配置：
  - 3 个专用 Master 节点
  - N 个 Data 节点（根据数据量）
  - 2 个副本
  - 多数据中心部署（可选）
```

## 分布式架构的优势与挑战

### 优势

1. **高可用**：节点故障不影响服务
2. **可扩展**：增加节点即可扩容
3. **高性能**：并行处理查询和索引
4. **负载均衡**：请求自动分发

### 挑战

1. **复杂性**：集群管理、故障诊断复杂
2. **网络依赖**：网络延迟和分区影响性能
3. **数据一致性**：分布式环境下的一致性保证
4. **分片规划**：分片数量难以调整

## 实战：理解分片分配

### 查看分片分配

```bash
GET /_cat/shards/products?v

index    shard prirep state   docs  store ip            node
products 0     p      STARTED 1000  10mb  192.168.1.100 node-1
products 0     r      STARTED 1000  10mb  192.168.1.101 node-2
products 1     p      STARTED 1200  12mb  192.168.1.101 node-2
products 1     r      STARTED 1200  12mb  192.168.1.102 node-3
products 2     p      STARTED 800   8mb   192.168.1.102 node-3
products 2     r      STARTED 800   8mb   192.168.1.100 node-1
```

**字段说明**：
- **prirep**：p（主分片）、r（副本分片）
- **state**：STARTED（正常）、INITIALIZING（初始化）、RELOCATING（迁移）
- **docs**：文档数量
- **store**：存储大小

### 手动分配分片

```json
POST /_cluster/reroute
{
  "commands": [
    {
      "move": {
        "index": "products",
        "shard": 0,
        "from_node": "node-1",
        "to_node": "node-3"
      }
    }
  ]
}
```

## 总结

**集群、节点、分片的关系**：
- **集群**：多个节点的集合，提供统一的索引和搜索服务
- **节点**：集群中的单个服务器，承担不同角色（Master、Data、Coordinating）
- **分片**：索引的物理分割，实现数据分布和并行处理

**分片路由算法**：
```
shard_num = hash(_routing) % num_primary_shards
```
- 主分片数量不可修改的原因
- 自定义路由优化查询

**高可用机制**：
- 主分片与副本分片分离存储
- Master 节点故障时自动选举
- 配置 quorum 避免脑裂

**扩展策略**：
- **水平扩展**：增加节点数量
- **垂直扩展**：升级节点硬件
- 合理规划分片数量
