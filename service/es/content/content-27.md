# 硬件选型与容量规划

## 概述

合理的硬件选型和容量规划是保障 Elasticsearch 性能和稳定性的基础。本章介绍 CPU、内存、磁盘、网络的选择标准以及容量评估方法。

## CPU 选择

### 核心数 vs 频率

**ES 的 CPU 需求特点**：

```
写入操作：
  - CPU 密集型（JSON 解析、分词、索引构建）
  - 多核并行处理

查询操作：
  - CPU + I/O 混合
  - 聚合计算 CPU 密集

推荐：
  - 优先选择多核心（16-32 核）
  - 频率 2.4GHz+ 即可
```

### CPU 配置建议

**节点类型与 CPU 配置**：

```
Master 节点：
  - 2-4 核心
  - 轻量计算
  
Data 节点：
  - 8-16 核心（中型）
  - 16-32 核心（大型）
  - 写入和查询密集
  
Coordinating 节点：
  - 4-8 核心
  - 聚合结果合并
```

### CPU 选型案例

```
小型集群（< 1TB）：
  - Intel Xeon E5-2670（8 核）
  - AMD EPYC 7302（16 核）

中型集群（1-10TB）：
  - Intel Xeon Gold 6130（16 核）
  - AMD EPYC 7402（24 核）

大型集群（> 10TB）：
  - Intel Xeon Platinum 8280（28 核）
  - AMD EPYC 7742（64 核）
```

## 内存规划

### 堆内存 vs 文件系统缓存

**内存分配原则**：

```
总内存：64GB

最佳分配：
  - ES 堆内存：31GB（50%，不超过 32GB）
  - 文件系统缓存：31GB（50%，Lucene Segment）
  - 操作系统：2GB

原因：
  - Lucene 依赖文件系统缓存
  - 50/50 分配性能最优
```

### 堆内存配置规则

**核心原则**：

```
1. Xms = Xmx（避免动态调整）
2. 不超过物理内存的 50%
3. 不超过 32GB（压缩指针限制）
4. 推荐：16GB - 31GB
```

**压缩指针（Compressed OOPs）**：

```
堆内存 ≤ 32GB：
  - 使用压缩指针
  - 对象引用占用 4 字节
  
堆内存 > 32GB：
  - 禁用压缩指针
  - 对象引用占用 8 字节
  - 内存利用率降低

示例：
  30GB 堆：可用 ~30GB
  34GB 堆：可用 ~27GB（浪费）

结论：31GB 是最佳配置
```

### 内存配置案例

```
16GB 物理内存：
  - ES 堆：8GB
  - 系统缓存：7GB
  - OS：1GB
  - 适合：小型开发环境

32GB 物理内存：
  - ES 堆：16GB
  - 系统缓存：15GB
  - OS：1GB
  - 适合：小型生产环境

64GB 物理内存：
  - ES 堆：31GB
  - 系统缓存：31GB
  - OS：2GB
  - 适合：中型生产环境（推荐）

128GB 物理内存：
  - ES 堆：31GB
  - 系统缓存：95GB
  - OS：2GB
  - 适合：大型生产环境
```

## 磁盘选择

### SSD vs HDD

**性能对比**：

```
SSD：
  - 随机读写：100x 快于 HDD
  - IOPS：50000+ vs 100-200
  - 延迟：< 1ms vs 10-20ms
  - 成本：高
  - 推荐：生产环境必须使用

HDD：
  - 顺序读写：可接受
  - 随机读写：极慢
  - 成本：低
  - 适用：冷数据、归档数据
```

### RAID 配置

**RAID 选择**：

```
RAID 0（推荐）：
  - 条带化，无冗余
  - 最大化吞吐量
  - ES 自身有副本机制
  - 推荐用于 Data 节点

RAID 1：
  - 镜像，高冗余
  - 写入性能减半
  - 适用：Master 节点

RAID 5/6（不推荐）：
  - 写入性能差
  - 不适合 ES
```

### 磁盘容量规划

**容量计算**：

```
数据量评估：
  - 文档数：1 亿条
  - 单文档大小：5KB
  - 原始数据量：500GB
  - 索引膨胀系数：1.8
  - 索引存储：900GB
  - 副本数：1
  - 总存储需求：1.8TB

预留空间：
  - 建议使用率 < 80%
  - 磁盘容量 = 1.8TB / 0.8 ≈ 2.3TB
  - 推荐配置：3TB SSD
```

### 磁盘配置案例

```
小型集群（< 1TB）：
  - 500GB - 1TB SSD
  - SATA SSD 或 NVMe

中型集群（1-10TB）：
  - 2TB - 4TB SSD
  - NVMe SSD（推荐）

大型集群（> 10TB）：
  - 热节点：2TB NVMe SSD
  - 温节点：4TB SATA SSD
  - 冷节点：8TB HDD
```

## 网络带宽需求

### 网络带宽计算

**写入带宽**：

```
写入速率：10000 docs/s
单文档大小：2KB
吞吐量：20MB/s

副本同步：20MB/s × 1 = 20MB/s
总带宽需求：40MB/s

推荐网络：1Gbps（125MB/s）足够
```

**查询带宽**：

```
查询 QPS：1000
平均响应大小：10KB
吞吐量：10MB/s

推荐网络：1Gbps
```

**大型集群**：

```
节点数：20+
跨节点数据传输频繁
推荐网络：10Gbps（万兆网卡）
```

### 网络配置建议

```
小型集群（3-5 节点）：
  - 千兆网卡（1Gbps）

中型集群（5-20 节点）：
  - 千兆网卡
  - 考虑链路聚合

大型集群（20+ 节点）：
  - 万兆网卡（10Gbps）
  - 独立存储网络
  - 低延迟交换机
```

## 容量评估

### 数据量评估

**评估步骤**：

```
1. 估算文档数量
   - 预计总文档数
   
2. 估算单文档大小
   - 平均文档大小（KB）
   
3. 计算原始数据量
   - 文档数 × 单文档大小
   
4. 应用索引膨胀系数
   - 原始数据量 × 1.5-2.0
   
5. 计算副本存储
   - 索引存储 × (1 + 副本数)
   
6. 预留空间
   - 总存储需求 / 0.8
```

**示例计算**：

```
需求：
  - 文档数：5 亿条
  - 单文档大小：3KB
  - 副本数：1

计算：
  原始数据量 = 500,000,000 × 3KB = 1.5TB
  膨胀系数 = 1.8
  索引存储 = 1.5TB × 1.8 = 2.7TB
  副本存储 = 2.7TB × 1 = 2.7TB
  总存储需求 = 2.7TB + 2.7TB = 5.4TB
  预留空间 = 5.4TB / 0.8 = 6.8TB
```

### 查询 QPS 评估

**单节点查询能力**：

```
简单查询（term、range）：
  - 500-1000 QPS/节点

复杂查询（多条件、聚合）：
  - 200-500 QPS/节点

全文搜索（match、multi_match）：
  - 300-700 QPS/节点
```

**节点数量计算**：

```
需求：1000 QPS 复杂查询

计算：
  单节点能力：300 QPS
  节点数 = 1000 / 300 ≈ 4 节点（最低）
  
推荐配置：
  - 6-8 个 Data 节点（考虑高峰和冗余）
```

### 写入 TPS 评估

**单节点写入能力**：

```
简单文档（< 1KB）：
  - 10000-15000 docs/s

中等文档（1-5KB）：
  - 5000-10000 docs/s

大文档（> 5KB）：
  - 2000-5000 docs/s
```

**节点数量计算**：

```
需求：每秒写入 20000 条中等文档

计算：
  单节点能力：8000 docs/s
  节点数 = 20000 / 8000 ≈ 3 节点（最低）
  
推荐配置：
  - 5-6 个 Data 节点（考虑副本和冗余）
```

## 集群规模计算

### 分片数量规划

**单分片大小**：

```
推荐：20-50GB
理想：30GB

计算公式：
  主分片数 = 总数据量 / 单分片大小

示例：
  数据量：3TB
  单分片：30GB
  主分片数 = 3000GB / 30GB = 100

实际配置：
  考虑到数据增长，配置 100-120 个主分片
```

### 节点数量规划

**基于分片数量**：

```
每节点最大分片数：1000
总分片数 = 主分片数 × (1 + 副本数)

示例：
  主分片数：100
  副本数：1
  总分片数 = 100 × 2 = 200
  
最少节点数 = 200 / 1000 = 1
推荐节点数 = 5-10（考虑性能和可用性）
```

**基于数据量**：

```
单节点存储能力：2-4TB

示例：
  总数据量：6TB（含副本）
  单节点容量：3TB
  
节点数 = 6TB / 3TB = 2（最少）
推荐节点数 = 3-5
```

## 水平扩容方案

### 扩容触发条件

```
磁盘使用率：
  - > 80%：计划扩容
  - > 85%：立即扩容

CPU 使用率：
  - > 70%：考虑扩容

查询延迟：
  - P95 > 500ms：考虑扩容

写入拒绝率：
  - > 1%：立即扩容
```

### 扩容步骤

```bash
# 1. 准备新节点
# 安装 ES，配置 cluster.name

# 2. 配置节点角色
node.roles: [data]

# 3. 启动节点
./bin/elasticsearch -d

# 4. 验证节点加入
GET /_cat/nodes?v

# 5. 等待分片再平衡
GET /_cat/health?v

# 6. 监控再平衡进度
GET /_cat/recovery?v
```

## 成本优化建议

### 冷热分离

**架构设计**：

```
热节点（7 天内数据）：
  - 高性能 SSD
  - 16 核 CPU
  - 64GB 内存

温节点（7-30 天数据）：
  - 普通 SSD
  - 8 核 CPU
  - 32GB 内存

冷节点（> 30 天数据）：
  - HDD
  - 4 核 CPU
  - 16GB 内存
```

**成本节省**：

```
全 SSD 方案：
  - 10TB 数据
  - 成本：10TB × $300/TB = $3000

冷热分离方案：
  - 热数据 2TB SSD：2TB × $300/TB = $600
  - 冷数据 8TB HDD：8TB × $50/TB = $400
  - 总成本：$1000
  - 节省：67%
```

### 压缩与归档

```bash
# 使用最佳压缩
PUT /archive
{
  "settings": {
    "codec": "best_compression"
  }
}
# 节省磁盘 30%-50%

# 定期清理过期数据
DELETE /logs-2023-*
```

## 硬件配置方案

### 小型集群（< 1TB）

```
节点配置：
  - 3 个混合节点
  - 8 核 CPU
  - 16GB 内存
  - 500GB SSD
  - 千兆网卡

成本：约 $5000-$8000

适用场景：
  - 中小型应用
  - 日志量 < 100GB/天
  - QPS < 500
```

### 中型集群（1-10TB）

```
节点配置：
  Master 节点（3 个）：
    - 2 核 CPU、4GB 内存、100GB SSD
  
  Data 节点（6 个）：
    - 16 核 CPU、64GB 内存、2TB SSD
    - 千兆或万兆网卡
  
  Coordinating 节点（2 个，可选）：
    - 8 核 CPU、16GB 内存

成本：约 $30000-$50000

适用场景：
  - 企业级应用
  - 日志量 100-500GB/天
  - QPS 1000-5000
```

### 大型集群（> 10TB）

```
节点配置：
  Master 节点（3 个）：
    - 4 核 CPU、8GB 内存、200GB SSD
  
  Hot Data 节点（10 个）：
    - 32 核 CPU、128GB 内存、4TB NVMe SSD
    - 万兆网卡
  
  Warm Data 节点（10 个）：
    - 16 核 CPU、64GB 内存、8TB SATA SSD
  
  Cold Data 节点（若干）：
    - 8 核 CPU、32GB 内存、16TB HDD

成本：约 $100000+

适用场景：
  - 海量数据平台
  - 日志量 > 1TB/天
  - QPS > 10000
```

## 总结

**CPU 选择**：
- 核心数 vs 频率：优先多核
- Data 节点：16-32 核
- Master 节点：2-4 核

**内存规划**：
- 堆内存：不超过 31GB
- 50% 堆内存 + 50% 系统缓存
- 推荐：64GB 物理内存

**磁盘选择**：
- SSD vs HDD：生产必须 SSD
- RAID 0：最大化性能
- 容量：预留 20% 空间

**网络带宽**：
- 小型集群：1Gbps
- 大型集群：10Gbps

**容量评估**：
- 数据量：文档数 × 大小 × 膨胀系数
- QPS：单节点 300-500
- TPS：单节点 5000-10000

**成本优化**：
- 冷热分离（节省 50%-70%）
- 压缩归档
- 定期清理

**下一步**：学习集群监控与运维，建立完善的监控体系。
