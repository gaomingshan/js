## 2. 协议性能特征对比

### 2.1 延迟对比

| 协议 | 典型延迟 | 延迟来源 |
|------|---------|---------|
| UDP | 最低 | 无连接建立、无拥塞控制 |
| TCP | 中等 | 三次握手 + 拥塞控制慢启动 |
| TLS 1.2 over TCP | 较高 | TCP 握手 + TLS 握手 (2-RTT) |
| TLS 1.3 over TCP | 中等 | TCP 握手 + TLS 握手 (1-RTT) |
| QUIC (首次) | 中等 | 合并握手 (1-RTT) |
| QUIC (0-RTT) | 最低 | 恢复连接无额外延迟 |

### 2.2 吞吐量对比

| 协议 | 吞吐量 | 瓶颈 |
|------|--------|------|
| TCP | 受拥塞控制限制 | 慢启动、丢包降速 |
| UDP | 无限制 | 应用层需自实现流控 |
| HTTP/1.1 | 6 连接并行 | 队头阻塞 |
| HTTP/2 | 单连接多路复用 | TCP 层队头阻塞 |
| HTTP/3 | 单连接无 HOL | QUIC 独立流重传 |

### 2.3 头部开销对比

| 协议 | 头部大小 | 说明 |
|------|---------|------|
| TCP | 20-60 字节 | 固定 + 选项 |
| UDP | 8 字节 | 固定 |
| TLS Record | 5 字节 | 记录层 |
| HTTP/1.1 | 数百-数千字节 | 文本头部，无压缩 |
| HTTP/2 | 1-数十字节 | HPACK 压缩 |
| MQTT | 2+ 字节 | 极简固定头 |
| gRPC | 5 字节前缀 | Length-Prefixed |

## 3. 安全性对比

| 协议 | 加密 | 认证 | 完整性 | 前向保密 |
|------|------|------|--------|---------|
| HTTP | ❌ | ❌ | ❌ | ❌ |
| HTTPS (TLS) | ✅ | ✅ | ✅ | ✅ (ECDHE) |
| SSH | ✅ | ✅ (公钥/密码) | ✅ | ✅ (DH) |
| FTPS | ✅ | ✅ | ✅ | 取决于配置 |
| MQTTS | ✅ | ✅ (TLS) | ✅ | ✅ (ECDHE) |
| AMQPS | ✅ | ✅ (TLS) | ✅ | ✅ (ECDHE) |
| SNMPv3 | ✅ (authPriv) | ✅ (USM) | ✅ | ❌ |
| gRPC | ✅ (TLS) | ✅ (TLS/Token) | ✅ | ✅ (ECDHE) |

## 4. 典型应用场景与协议选择

### 4.1 Web 应用

```
静态网站:     HTTP/2 + TLS (CDN 加速)
动态网站:     HTTP/2 + TLS (多路复用)
实时通知:     WebSocket over TLS (全双工)
低延迟 Web:   HTTP/3 + QUIC (0-RTT, 无 HOL)
```

### 4.2 微服务通信

```
同步调用:     gRPC (Protobuf, 流式, 代码生成)
异步消息:     AMQP (可靠传递, 路由)
服务发现:     DNS + gRPC (健康检查)
```

### 4.3 IoT 与嵌入式

```
受限设备:     MQTT (低开销, QoS, 遗嘱)
传感器网络:   MQTT-SN (UDP, 主题 ID)
CoAP:        受限应用协议 (UDP, 类 REST)
```

### 4.4 流媒体

```
视频点播:     HTTP(S) + HLS/DASH (CDN 友好)
实时音视频:   RTP/RTCP + SRTP (WebRTC)
流控制:       RTSP (播放控制)
```

### 4.5 邮件系统

```
发送:         SMTP + STARTTLS (端口 587)
接收:         IMAP + TLS (多设备同步)
```

### 4.6 网络管理

```
设备监控:     SNMPv3 (authPriv)
配置管理:     NETCONF over SSH
```

## 5. 协议组合使用模式

### 5.1 HTTPS = HTTP + TLS + TCP

```
应用数据 → HTTP 请求/响应
         → TLS 加密记录
         → TCP 可靠传输
         → IP 路由
```

最经典的协议组合，Web 的基础。

### 5.2 gRPC = Protobuf + HTTP/2 + TLS + TCP

```
RPC 调用 → Protobuf 序列化
         → gRPC Length-Prefixed 帧
         → HTTP/2 DATA/HEADERS 帧
         → TLS 加密
         → TCP 传输
```

微服务间通信的标准组合。

### 5.3 WebRTC = SRTP + DTLS + ICE + UDP

```
音视频   → RTP/RTCP
         → SRTP 加密
         → UDP 传输
         → ICE (STUN/TURN) NAT 穿透
         → DTLS 密钥协商
```

浏览器实时通信的完整协议栈。

### 5.4 MQTTS = MQTT + TLS + TCP

```
IoT 消息 → MQTT 帧 (QoS/主题)
         → TLS 加密
         → TCP 传输
```

IoT 安全通信组合。

### 5.5 HTTP/3 = HTTP + QUIC + UDP

```
Web 请求 → HTTP/3 语义
         → QUIC 帧 (多路复用)
         → TLS 1.3 内置
         → UDP 传输
```

下一代 Web 协议组合。

## 6. 选型决策树

```
需要可靠传输?
├── 是 → 对延迟敏感?
│   ├── 是 → 需要多路复用?
│   │   ├── 是 → HTTP/2 + TLS 或 gRPC
│   │   └── 否 → TCP + TLS
│   └── 否 → 需要消息路由?
│       ├── 是 → AMQP
│       └── 否 → 需要流式传输?
│           ├── 是 → gRPC (streaming)
│           └── 否 → HTTP/1.1 + TLS
└── 否 → 对延迟极敏感?
    ├── 是 → 需要加密?
    │   ├── 是 → DTLS 或 QUIC
    │   └── 否 → UDP (RTP/MQTT-SN)
    └── 否 → 需要发布/订阅?
        ├── 是 → MQTT
        └── 否 → SNMP (管理) / DNS (解析)
```

## 7. 协议演进趋势

| 趋势 | 说明 |
|------|------|
| **HTTP/3 替代 HTTP/2** | QUIC 解决 TCP 队头阻塞 |
| **gRPC 替代 REST** | 微服务场景，性能和类型安全优势 |
| **MQTT 替代 HTTP for IoT** | 低开销、QoS、遗嘱消息 |
| **TLS 1.3 淘汰 TLS 1.2** | 更快握手、更强安全 |
| **DoH/DoT 替代明文 DNS** | 隐私保护 |
| **SNMPv3 替代 v1/v2c** | 安全合规 |
| **SFTP 替代 FTP/FTPS** | 单通道、防火墙友好 |
