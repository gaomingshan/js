# 前端代码规范与工程化 - 系统化学习大纲

> 面向后端开发者的前端工程治理体系

---

## 📖 学习目标

从**工程系统**的角度理解前端代码规范，建立类似后端架构规范、编码约束、工程治理的认知体系，理解前端代码在"多人协作 + 长期演进"中的失控风险与治理手段。

**核心类比**：
- 前端代码规范 ≈ 后端编码规范 / 架构约束
- Lint / Format ≈ 编译期检查 / 静态分析
- 工程约束 ≈ 服务治理与技术规范

## 核心要点总结

### 三个核心认知

1. **规范不是风格偏好，是工程约束**
   - 目标：可维护性、协作效率、技术债控制
   - 手段：自动化工具 + 团队文化

2. **工具是手段，不是目的**
   - ESLint：代码质量防线
   - Prettier：格式统一工具
   - TypeScript：类型安全保障

3. **规范需要演进，不是一成不变**
   - 新项目 vs 历史项目
   - 团队共识 vs 强制推行
   - 持续优化 vs 固守陈规

### 后端开发者的转变

从后端到前端，规范思维的迁移：

| 后端 | 前端 | 共同点 |
|------|------|--------|
| 编译期检查 | Lint / TypeScript | 静态分析 |
| 代码格式化工具 | Prettier | 格式统一 |
| 架构约束 | 规范 + 工具链 | 工程治理 |
| 服务治理 | Monorepo 治理 | 多项目管理 |
| 技术债管理 | 规范演进 | 持续改进 |
