你是一名资深前端架构师与技术教育者，在 @package-management 下，请生成前端「包管理器」系统化学习内容及面试题，要求如下：

【学习对象】
- 有 JavaScript / Node.js 基础，想系统理解前端包管理器机制的开发者（含后端经验者）
- 目标：理解包管理器的设计原理、依赖解析机制、版本控制策略与工程实践，能够在真实项目与团队中做出合理选型和配置

【生成顺序与文件结构】
1. 生成大纲文件 content-outline.md，每个小节带有 Markdown 跳转链接到对应 content 文件
2. 生成教学内容，存放于 content/ 目录
3. 生成面试题汇总，存放于 quiz/quiz.md 

【文件命名规则】
- 大纲文件：content-outline.md 
- 教学内容：content/content-1.md、content/content-2.md … 按大纲顺序递增
- 面试题汇总：quiz/quiz.md 
- 链接格式：[小节标题](./content/content-x.md)，x 按大纲顺序递增
- 统一使用 Markdown，中文，技术严谨且教学友好

【教学内容要求】
- 核心思想必须围绕：
  - 依赖模型与依赖图（dependency graph）
  - 语义化版本（semver）与版本解析规则
  - 安装算法与去重策略
  - 锁文件（lockfile）的设计与一致性保障
  - npm / yarn / pnpm 的设计取舍与差异
- 每小节必须包含：
  - 核心概念说明
  - 设计动机 / 工作原理（why & how）
  - 常见误区或边界问题
  - 前端工程或团队实践示例
- 精简但有深度，禁止命令清单式或百科式罗列
- 难度分级：基础 → 进阶 → 高级

【面试题要求】
- 单选 / 多选 / 代码题 / 场景题结合，约 100 题
- 难度递进：基础 → 原理 → 综合 / 架构决策
- 覆盖范围必须与教学大纲一致
- 题目强调“为什么这么设计”“不同方案如何取舍”
- 每题提供答案与解析
- 答案解析要求：
  - 分层呈现：结论 → 原理 → 易错点 / 反例
  - 避免大段堆砌文字，保证 Markdown 可读性
- 面试题仅生成一个汇总文件 quiz/quiz.md 

【质量红线】
- 禁止只教命令、不讲原理
- 禁止简单对比 npm / yarn / pnpm 表面差异
- 禁止照搬官方文档或 changelog
- 概念讲不清时应缩减内容，而不是堆字
- 面试题解析超过 15 行 Markdown 必须拆分

【额外说明】
- 明确对比前端与后端依赖管理（如 Maven / Gradle / Go modules）
- 教学内容需覆盖真实工程问题：
  - 依赖冲突
  - 幽灵依赖
  - 安装不一致
  - CI / Monorepo 场景
- 大纲文件必须生成 Markdown 链接，格式为 [小节标题](./content/content-x.md)，确保点击可跳转
